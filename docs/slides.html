§1 Introduction

Problem Statement & Motivation

Context

Wikiquote is one of the largest open-source quote repositories, containing over 1.3 million quotations from historical figures, literary works, and cultural sources. However, it lacks any form of voice-based access, speaker-aware personalization, or conversational search interface.


Problem

  • Existing quote retrieval systems are text-only — no support for spoken queries
  • No system combines ASR + speaker biometrics + TTS for personalized quote search
  • Classical and literary quotes contain archaic language that standard ASR models handle poorly
  • Multi-user scenarios require manual login; voice-based identification is a more natural alternative

Research Questions

RQ1. Can a domain-prompted Whisper ASR model accurately transcribe literary and archaic speech for downstream quote retrieval?

RQ2. Can ECAPA-TDNN speaker embeddings reliably distinguish enrolled users from short (3–5 second) voice samples in a real-time web application?

RQ3. Does the combined end-to-end pipeline — from voice input to personalized TTS response — deliver a satisfactory user experience within acceptable latency constraints on consumer-grade hardware?

§2 Related Work

Related Work & Technology Landscape

Automatic Speech Recognition

OpenAI's Whisper (Radford et al., 2023) achieves near-human-level transcription accuracy across 99 languages. We adopt the base model (74M parameters) via faster-whisper, a CTranslate2-optimized implementation achieving up to 4× speedup over the reference PyTorch model. Domain-specific initial prompts have been shown to reduce Word Error Rate (WER) on domain-specific vocabulary (Koenecke et al., 2020).


Text-to-Speech Synthesis

gTTS wraps Google's neural TTS API, providing natural prosody without local GPU requirements. Coqui TTS offers the VITS architecture for local, multi-speaker synthesis. We use both to provide online and offline TTS capabilities with per-user voice personalization.

Speaker Verification

ECAPA-TDNN (Desplanques et al., 2020) extends the conventional TDNN architecture with Squeeze-Excitation blocks, multi-layer feature aggregation, and channel-dependent attention. It is the current state-of-the-art for short-utterance speaker verification. SpeechBrain provides a pre-trained model producing 192-dimensional x-vectors, trained on VoxCeleb1 + VoxCeleb2 (7,000+ speakers).


Graph Databases for NLP

Neo4j naturally models the relationships between persons, works, and quotations as a labeled property graph. Full-text indexing via Apache Lucene enables sub-100ms BM25 search over millions of text nodes — an order of magnitude faster than equivalent SQL LIKE queries on unstructured text.

§3 System Architecture

High-Level Architecture

The system follows a two-phase pipeline architecture. Phase 1 handles input understanding (transcription and speaker identification). Phase 2 handles retrieval and response generation. A human-in-the-loop review step bridges the two phases.

Phase 1 — Transcription & Speaker ID

  • Audio captured via MediaRecorder API in the browser (streamlit-audiorecorder)
  • faster-whisper transcribes with a literary domain prompt to improve accuracy on archaic vocabulary
  • SpeechBrain ECAPA-TDNN extracts 192-dim x-vector and matches against enrolled speaker profiles
  • Both processes run on the same audio file — no duplicate recording required

Phase 2 — Retrieval & Response

  • User reviews and optionally corrects the transcript in an editable text area
  • Rule-based intent router classifies query: quote-text vs. author vs. topic
  • Neo4j full-text index (Lucene BM25) returns top-5 ranked quotes with similarity scores
  • Best match synthesized via gTTS using the identified speaker's voice preferences

§4 Data Engineering

Wikiquote Corpus & Neo4j Graph Model

Data Source & Ingestion Pipeline

The dataset is derived from the English Wikiquote XML dump (December 2025). A custom SAX-based XML parser extracts quote text, attributed author, and source work. The ingestion pipeline processes records in batches of 1,000 via Neo4j's Bolt protocol.


Quality Filtering

The raw corpus contains significant noise — movie dialogue, screenplay fragments, bibliographic citations, and parody quotes. We apply 15+ regex filters:

  • Minimum text length: 120 characters
  • Exclude page references (p. \d+), URLs, bracket annotations
  • Exclude excessive punctuation (!.*!.*!), incomplete text (&c.)
  • Near-duplicate removal at cosine similarity threshold 0.95

Domain Prompt Strategy

"Quote search. The user may say a famous literary quote, an author name, a work title, or a topic. Literary and poetic language may appear."

This initial_prompt parameter biases Whisper's language model toward literary vocabulary — reducing Word Error Rate on archaic terms like thee, wherefore, doth, hath that are common in Wikiquote but rare in general speech corpora.

Processing Pipeline

1

2

3

4

5

6

Design Decision: Human-in-the-Loop

Rather than silently passing ASR output to search, we deliberately show the transcript to the user for correction. This is critical for literary quotes where a single misheard word can completely change the search result.

§5.2 Speaker Identification

Speaker Identification — ECAPA-TDNN

Model Architecture

ECAPA-TDNN (Emphasized Channel Attention, Propagation and Aggregation in TDNN) extends the standard Time-Delay Neural Network with three key innovations:

  • Squeeze-Excitation (SE) blocks — channel-wise attention to emphasize speaker-discriminative frequency bands
  • Multi-layer feature aggregation (MFA) — combines outputs from all TDNN layers, capturing both local and global temporal patterns
  • Channel-dependent attention — attentive statistics pooling across the time axis

Training Data

The SpeechBrain pre-trained model was trained on VoxCeleb1 + VoxCeleb2 (7,000+ speakers, millions of utterances from YouTube interviews). The model generalizes well to clean microphone recordings.

Identification Pipeline

1
Enrollment — User records 3–5 sec sample → 192-dim x-vector extracted → stored as Speaker node in Neo4j
2
Extraction — On each query, extract x-vector from the same audio used for ASR
3
Matching — Compute cosine similarity against all enrolled embeddings
4
Decision — If max similarity ≥ τ = 0.55, return speaker name + confidence; else return "Guest"

Per-Speaker Personalization

  • Voice gender — male / female / neutral, stored per speaker profile in Neo4j
  • Speed & accent — configurable per user (e.g. British, American)
  • Automatic loading — preferences loaded on speaker identification, fallback to defaults for Guest

Audio-Text Validation

Problem: Risk of TTS speaking different content than the displayed quote (wrong index, race condition, or caching artifact).

Solution: The orchestrator calls _validate_audio_text() which performs an exact string match between the quote shown on screen and the text passed to the TTS engine.

Result: Zero audio-text mismatches observed in testing. Any mismatch raises an exception and logs a detailed error for debugging.


Output

  • Audio plays inline in the browser via <audio> element
  • Downloadable WAV file for each synthesized quote
§6 Retrieval & Ranking

Intent Routing & Neo4j Full-Text Search

Intent Classification (Rule-Based)

A lightweight, deterministic regex router classifies each query. The design choice to avoid ML-based classifiers here is deliberate — zero latency, fully interpretable, no external API dependency.


PatternDetected IntentExample
^(quotes? )?by [A-Z]...Author search"by Einstein"
^what did [A-Z]... sayAuthor search"what did Gandhi say"
^(find )?from [A-Z]...Work search"from Hamlet"
(default)Quote-text search"to be or not to be"

Author names are resolved via difflib.SequenceMatcher with fuzzy edit distance ≤ 2 (e.g. "Shakespear" → "William Shakespeare").

Ranking Algorithm

CALL db.index.fulltext.queryNodes(
  'quoteIndex', $query
) YIELD node AS q, score
MATCH (p:Person)-[:SAID]->(q)
WHERE size(q.text) >= 120
  AND NOT q.text =~ '.*\\[.*\\].*'
  AND NOT q.text =~ '.*p\\.\\s*\\d+.*'
RETURN q.text, p.name,
  score * q.quality_score AS rank
ORDER BY rank DESC LIMIT 5

Quality Scoring Factors

  • Boost: canonical quotes, famous authors (Shakespeare, Einstein, Gandhi), longer text
  • Penalize: citations, stage directions, duplicate entries, low quality_bucket
  • Final score = BM25 × quality_score → normalized to percentage for display

§8.1 Challenges

Technical Challenges & Solutions

⚠️ Data Quality Noise

Problem: The corpus contains movie dialogue ("Code 10! Code 10!"), screenplay fragments ([picket sign] COGITO ERGO NOTHING!), bibliographic citations ("Thomas Paine, in Life and Writings… p. 13").

Solution: 15+ regex filters applied during ingestion and query-time. Minimum text length set to 120 characters.

Status: Partially Resolved — some noise persists. An ML-based quality classifier would be the ideal solution.

🔍 Missing Canonical Quotes

Problem: Descartes' "Cogito ergo sum" and several Socrates quotes are absent from the database.

Root Cause: The Wikiquote XML parser encountered non-standard page formatting. The database has references to these figures but not quotes by them.

Status: Unresolved — data limitation, not code defect.

⏱️ CPU Latency Bottleneck

Problem: End-to-end pipeline takes 6–10 seconds on CPU. Primary bottlenecks: Whisper ASR (~3s) and Coqui TTS (~3s).

Mitigations applied:

  • faster-whisper CTranslate2 backend (4× vs. reference)
  • Model caching via Streamlit session_state
  • gTTS cloud API as primary TTS (faster than local Coqui)

Projected: GPU deployment would reduce total to <2 seconds.

📝 ASR on Archaic Language

Problem: Whisper struggles with literary vocabulary — "wherefore art thou" → "where for art thou".

Solutions:

  • Literary domain prompt injected via initial_prompt
  • Editable transcript review before search — users correct errors manually
§8.2 Future Directions

Limitations & Future Work

Data Improvements

  • Re-parse Wikiquote XML with stricter quote boundary detection
  • Train a supervised quote quality classifier (BERT-based binary: real quote vs. noise)
  • Integrate supplementary sources (BrainyQuote, Goodreads Quotes)
  • Add Neo4j uniqueness constraints to prevent duplicate entries
  • Implement user-driven feedback loop for quality rating

Model Upgrades

  • Upgrade ASR to Whisper medium or large-v3 for improved archaic text accuracy
  • Fine-tune ECAPA-TDNN on literary reading speech domain data
  • Implement speaker adaptation — update embeddings with successive voice samples
  • Explore voice cloning for truly personalized TTS synthesis
  • GPU inference deployment (CUDA / MPS) for <2s latency

System Extensions

  • Semantic search via sentence-transformer embeddings (beyond keyword matching)
  • Multi-language support — French, German, Arabic Wikiquote editions
  • REST API layer for third-party integration
  • Progressive Web App (PWA) for mobile access
  • Real-time streaming ASR (instead of record-then-transcribe)

Conclusion

Summary & Contributions

This project demonstrates a complete, end-to-end, multi-user voice-enabled NLP system that bridges speech input, knowledge retrieval, and personalized audio output. The system integrates state-of-the-art models — Whisper for ASR, ECAPA-TDNN for speaker identification — with a Neo4j graph database containing 1.3M+ quotes and a real-time Streamlit web interface, operating entirely on consumer-grade CPU hardware.
