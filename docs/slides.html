Section 1 · Motivation

Motivation & Problem Statement

The Problem

  • Wikiquote contains 1.3M+ quotes but no voice-accessible search interface
  • Existing quote retrieval systems are text-only — no speech input, no speaker awareness
  • Multi-user systems typically require manual login; voice biometrics can replace this
  • TTS responses are generic — users want a personalized voice experience

Research Questions

RQ1. Can Whisper ASR accurately transcribe literary and archaic speech for quote retrieval?
RQ2. Can ECAPA-TDNN speaker embeddings reliably identify enrolled users from short voice samples?
RQ3. Does the combined pipeline provide a satisfactory end-to-end user experience within acceptable latency?

Section 4 · ASR

Speech Recognition — Whisper ASR

Model Details

  • faster-whisper: CTranslate2-optimized Whisper
  • Model size: base (74M params) — balance of speed & accuracy
  • Latency: ~2–3 seconds on CPU (Apple Silicon)
  • Input: WAV audio from browser or file upload
  • Output: plain-text transcript + language detection

Domain Prompt Engineering

"Quote search. The user may say a famous literary quote, an author name, a work title, or a topic. Literary and poetic language may appear."

This biases the language model toward archaic vocabulary (thee, wherefore, doth), significantly reducing transcription errors on classical text.

Processing Flow

① Audio bytes → temporary WAV file
② faster-whisper loads model (cached after first run)
③ Transcribe with domain initial_prompt
④ Return text + confidence segments
⑤ Display in editable textarea for user correction
faster-whisper CTranslate2 domain prompting CPU ~2–3s
Section 5 · Speaker ID

Speaker Identification — ECAPA-TDNN

Model Architecture

  • ECAPA-TDNN (Emphasized Channel Attention, Propagation and Aggregation — Time Delay Neural Network)
  • Implemented via SpeechBrain framework
  • Produces 192-dimensional x-vectors per utterance
  • Trained on VoxCeleb2 dataset (millions of speaker samples)
  • Robust to short utterances (3–5 sec) and background noise

Enrollment Process

  • User records 3–5 second voice sample
  • Embedding extracted and stored in Neo4j as Speaker node
  • Voice preferences (gender, speed, accent) stored alongside

Identification Algorithm

① Extract x-vector from query audio
② Load all enrolled speaker embeddings from Neo4j
③ Compute cosine similarity for each pair
④ Best match score vs. threshold τ = 0.55
⑤ Return speaker name + confidence, or "Guest"

Section 6 · Retrieval

Intent Routing & Neo4j Retrieval

Rule-Based Intent Router

  • Lightweight regex pattern classifier — zero latency, fully interpretable
  • Priority: Quote-text first → Author → Topic fallback
  • Author patterns: ^by [Name], what did [Name] say, from [Work]
  • Fuzzy author resolution via difflib.SequenceMatcher (edit distance ≤ 2)

Ranking Algorithm

  • Neo4j full-text Lucene index on Quote.text
  • Base score from BM25 full-text similarity
  • Boosted for: canonical quotes, famous authors, longer text
  • Penalized for: citations, duplicates, low quality_score
  • Top-5 results returned with normalized similarity %

Sample Cypher Query

CALL db.index.fulltext.queryNodes(
  'quoteIndex', $query
) YIELD node AS q, score
MATCH (p:Person)-[:SAID]->(q)
WHERE size(q.text) >= 120
  AND NOT q.text =~ '.*p\\.\\s*\\d+.*'
  AND NOT q.text =~ '.*https?://.*'
RETURN q.text, p.name,
       score * q.quality_score AS rank
ORDER BY rank DESC
LIMIT 5

Section 8 · Challenges

Technical Challenges & Solutions

⚠️ Data Quality Noise

Problem: ~1.3M entries include movie dialogue, screenplay fragments ([picket sign] COGITO ERGO NOTHING!), bibliographic citations (p. 13), and parody quotes.

Solution: 15+ regex filters (min length 120, no page refs, no brackets, no URLs). Partially resolved — some noise persists.

🔍 Missing Canonical Quotes

Problem: Descartes' "Cogito ergo sum", several Socrates quotes are absent. Database has references to these figures but not their actual quotes.

Root Cause: Non-standard XML formatting in Wikiquote source pages. A data-ingestion gap, not a retrieval failure.

⏱️ CPU Latency

Problem: Full pipeline takes 6–10 seconds on CPU — ASR (~3s) + Speaker ID (~1s) + TTS (~2s).

Mitigation: faster-whisper (4× speedup over base Whisper), model caching in session_state. GPU would reduce total to <2s.

📝 ASR Accuracy on Archaic Text

Problem: Standard ASR models struggle with literary language — "wherefore art thou" transcribed as "where for art thou".

Solution: Literary domain prompt + editable transcript review (human-in-the-loop correction before search).

Section 8 · Future Work

Limitations & Future Directions

Data Quality

  • Re-parse Wikiquote XML with stricter extraction rules
  • Train ML-based quote quality classifier
  • Integrate supplementary quote databases (BrainyQuote, QuoteDB)
  • Add Neo4j uniqueness constraint to prevent duplicates

Model Improvements

  • Upgrade to Whisper medium or large-v3 for better archaic text accuracy
  • Fine-tune ECAPA-TDNN on literary reading speech domain
  • Add speaker adaptation — update embeddings with more samples over time
  • GPU deployment to reduce latency below 2s

System Extensions

  • Semantic search using sentence-transformer embeddings
  • Multi-language support (French, Arabic Wikiquote)
  • User feedback loop to rate quote quality
  • REST API layer for third-party integration
  • Mobile-friendly PWA version
Conclusion

Summary

This project successfully demonstrates a complete end-to-end voice-enabled NLP pipeline — from raw speech input to personalized audio response — operating entirely on consumer hardware. The system combines state-of-the-art ML models (Whisper, ECAPA-TDNN) with a graph database (Neo4j) and a real-time web interface (Streamlit) to provide a novel multi-user quote retrieval experience.
