🧩 System Modules

What the System Does

The project is built around four tightly integrated modules. Each module solves a distinct challenge in the voice-to-quote pipeline. Together they form a seamless, multi-user NLP system that bridges voice interaction, knowledge retrieval, and personalized audio output.

🤖

Chatbot & Live Search

The chatbot interface is the primary text entry point. It provides smart one-click suggestion pills across four categories — Authors, Works, Moods, and Topics — so users can explore the database without knowing exactly what to type. Under the hood, every keystroke triggers a Neo4j full-text query and updates results in real time without a page reload.

  • Suggestion pills: Authors, Works, Moods, Topics
  • Live autocomplete — results update as you type (min 3 chars)
  • Top-5 ranked results with normalized similarity scores
  • Conversational intent routing: quote text → author → topic fallback
  • Anti-hallucination response generator — only shows DB results
🎤

Speaker Identification

Users can enroll their voice by recording a 3–5 second sample. The system extracts a 192-dimensional voice embedding using SpeechBrain's ECAPA-TDNN model and stores it in the Neo4j graph. On future interactions, any recorded voice is matched against all enrolled profiles using cosine similarity, identifying the speaker with 81.5% accuracy even in noisy conditions.

  • SpeechBrain ECAPA-TDNN — state-of-the-art speaker embeddings
  • 192-dimensional speaker vectors stored per user in Neo4j
  • Cosine similarity matching, threshold = 0.55
  • Multi-user profile management (enroll, identify, delete)
  • Graceful fallback to "Guest" if speaker is unknown
🔊

Text-to-Speech

Each enrolled speaker can configure a preferred voice profile — male, female, or neutral — along with speaking speed and regional accent. When the system returns a quote, it synthesizes audio using the speaker's stored preferences, creating a personalized audio response. The TTS output is validated against the displayed text to guarantee synchronization between what is shown and what is spoken.

  • gTTS (Google TTS) as primary engine — natural prosody
  • Coqui VITS multi-speaker model as alternative
  • macOS say as offline fallback
  • Voice preferences stored per speaker profile
  • Audio validation: TTS text must match displayed quote exactly
  • In-browser playback + downloadable audio file
🎙️

Full Voice Pipeline

The voice chat module combines all other modules into a single seamless flow. The user records their voice, Whisper transcribes it with a literary domain prompt to improve accuracy on archaic language, the speaker is identified, and the transcript is shown in an editable box so errors can be corrected before searching. The system then retrieves the top-5 matching quotes and plays back the best match in the speaker's personalized voice.

  • faster-whisper base model — 2–3s transcription on CPU
  • Domain prompt: literary, archaic, and poetic language aware
  • Editable transcript review before search — corrects ASR errors
  • Simultaneous speaker ID and transcription in one step
  • "Record Again" resets voice state without losing app session
🔄 Step-by-Step Process

The Voice Pipeline — End to End

Follow a single query from the moment the user presses Record, all the way through to a personalized spoken response. Every step is explained below with the technology used and the design decision behind it.

01

🎤 Voice Recording

The user presses the ⏺ Record button inside the browser. Audio is captured in real time using the streamlit-audiorecorder widget, which wraps the browser's native MediaRecorder API. The result is a WAV audio segment in memory — no file is written to disk at this stage. Users can also upload a pre-recorded file (WAV, MP3, OGG, M4A) to bypass the microphone.

streamlit-audiorecorder WAV / MP3 / OGG MediaRecorder API
02

🌐 Speech-to-Text (Whisper ASR)

The WAV audio is written to a temporary file and passed to faster-whisper running the base model. A custom domain prompt is injected — "Quote search. The user may say a famous literary quote, an author name, a work title, or a topic. Literary and poetic language may appear." — which biases the language model toward archaic vocabulary (e.g. "wherefore", "thee", "doth") and significantly improves transcription accuracy on classical text. The model returns a plain-text transcript in ~2–3 seconds on CPU.

faster-whisper Whisper base domain prompt ~2–3s latency
03

🧠 Speaker Identification (ECAPA-TDNN)

In parallel with transcription, SpeechBrain's ECAPA-TDNN model extracts a 192-dimensional voice embedding from the same audio file. This embedding is compared against all enrolled speaker profiles stored in Neo4j using cosine similarity. If the best match exceeds a 0.55 threshold, the speaker is identified and their name and voice preferences are loaded for personalized output. If no match is found, the session continues as "Guest" with a default voice.

SpeechBrain ECAPA-TDNN 192-dim embedding cosine similarity threshold 0.55
04

✏️ Editable Transcript Review

The ASR transcript is shown to the user in an editable text area before any search is performed. This is a deliberate UX design decision: Whisper is not perfect, especially on archaic literary language, and allowing the user to fix errors before searching dramatically improves retrieval quality. The speaker identity is also displayed with a confidence score. The user then clicks "Search with this text" to proceed.

editable textarea speaker name + confidence human-in-the-loop
05

🔍 Intent Routing & Neo4j Search

The query is classified by a rule-based intent router using regex patterns. The router first checks if the query matches an explicit author request (e.g. "quotes by Einstein", "what did Shakespeare say") — if so, it searches by Person name. Otherwise it defaults to quote-text-first search, querying the Neo4j full-text index on Quote.text. Results are ranked by a quality scoring function that boosts canonical quotes from famous authors (Shakespeare, Gandhi, Einstein), penalizes citations and duplicates, and returns the top-5 results with normalized similarity percentages.

regex intent router Neo4j full-text index quality ranking top-5 results
06

🔊 Personalized TTS Response

The best-matching quote is synthesized using the identified speaker's preferred TTS voice. The system calls gTTS (Google Text-to-Speech) with the speaker's configured language, speed, and accent settings. Before audio is played, a validation step confirms that the text passed to TTS exactly matches the quote displayed on screen — preventing any mismatch between what is shown and what is spoken. The audio plays inline in the browser and can be downloaded as a WAV file.

gTTS Google TTS Coqui VITS audio validation per-speaker voice prefs
🛠️ Technologies

Tech Stack

Each technology was chosen for a specific reason. The stack covers the full spectrum from data storage and retrieval, through machine learning models for ASR and speaker ID, to the frontend and audio synthesis layer.

Frontend
Streamlit
Rapid Python-native web UI with session state management and component reuse. streamlit-audiorecorder enables real-time microphone capture inside the browser.
✦ Chosen for fast iteration and tight Python integration
ASR
faster-whisper
CTranslate2-optimized Whisper inference — up to 4× faster than the original. Base model balances speed and accuracy. Domain prompting compensates for archaic literary vocabulary.
✦ Best open-source ASR for general speech on CPU
Speaker ID
SpeechBrain
ECAPA-TDNN model extracts 192-dim x-vectors that capture the unique acoustic signature of each speaker. Robust to background noise and short utterances (3–5 sec).
✦ State-of-the-art speaker verification model
TTS
gTTS / Coqui
Google TTS provides natural, fluent speech synthesis with multilingual support. Coqui VITS is used as a local multi-speaker alternative. macOS say as offline fallback.
✦ Natural prosody + per-user voice personalization
Database
Neo4j
Graph database with native full-text search indexes on 1.3M+ quotes. The graph schema models Person→Quote, Person→Work, and Work→Quote relationships, enabling rich author-attribution queries.
✦ Graph model naturally fits quote–author–work relationships
NLP Routing
Custom Router
Lightweight regex-based intent classifier that detects quote-text vs. author vs. topic queries. difflib fuzzy matching resolves author name variations (e.g. "Wilde" → "Oscar Wilde").
✦ Zero latency, fully interpretable, no external API

⚠️

Data Quality Noise

The Wikiquote XML dump contains ~1.3M entries, but a significant portion are movie dialogue, screenplay fragments, bibliographic citations, and parody quotes. 15+ regex filters were applied (length minimum 120 chars, no page numbers, no bracket stage directions, no URLs), but some noise still passes through. A dedicated ML quality classifier would be the ideal solution.

🔍

Missing Canonical Quotes

Paradoxically, some of the most famous philosophical quotes are absent from the database. Descartes' "Cogito ergo sum" and several Socrates quotes are missing because the Wikiquote XML parser encountered non-standard formatting for those pages. The database has references to these authors but not their direct quotes — a data ingestion gap, not a retrieval problem.

⏱️

CPU Latency

The 6–10 second round-trip on CPU is the main UX limitation for a live demo. The two biggest contributors are Whisper ASR (~3s) and Coqui TTS (~3s). Switching to GPU inference and caching speaker embeddings in memory (rather than Neo4j) would reduce latency below 2 seconds, making the interaction feel natural and conversational.
